# Tooling

How the day-to-day commands hang together. Mostly Make + npm scripts; a few notes on what each tool is doing under the hood.

## Make is the entry point

Bare `make` shows every available target with a one-line description, grouped by section. The root `Makefile` is a thin include manifest; concerns live in `makefiles/<group>/<concern>.mk`. The layout mirrors the backend's so the two repos read the same way.

```
makefiles/
  vars.mk                 # NPM, NPX, SRC, DIST, COV_DIR
  help.mk                 # help + check meta-targets
  clean.mk
  install/deps.mk
  quality/lint.mk
  quality/test.mk
  dev/server.mk
  build/bundle.mk
  tooling/openapi.mk
```

Common targets:

| Target               | What it does                                               |
| -------------------- | ---------------------------------------------------------- |
| `make help`          | print the menu                                             |
| `make install`       | `npm ci` тАФ clean install from `package-lock.json`          |
| `make lock`          | regenerate `package-lock.json` without installing          |
| `make dev`           | vite dev server on `:5173`                                 |
| `make preview`       | serve the built bundle on `:4173`                          |
| `make build`         | `tsc -b && vite build` тАФ production bundle in `dist/`      |
| `make build-stats`   | build, then print per-asset sizes                          |
| `make lint`          | ESLint over the codebase                                   |
| `make format`        | Prettier-write everything, then `eslint --fix`             |
| `make typecheck`     | `tsc -b --noEmit`                                          |
| `make test`          | one-shot vitest run                                        |
| `make test-watch`    | vitest in watch mode                                       |
| `make test-cov`      | vitest with coverage                                       |
| `make gen-api`       | regenerate `src/api/schema.ts` from backend `openapi.yaml` |
| `make gen-api-check` | CI guard тАФ fails if committed schema is stale              |
| `make check`         | `lint typecheck test` тАФ the pre-push gate                  |
| `make clean`         | remove `dist/`, vite cache and coverage                    |
| `make clean-deps`    | destructive тАФ also removes `node_modules`                  |

`make check` is what you run before you `git push`. CI runs the same combo (plus `make build`) on every PR.

## Pre-commit hooks

We use **husky** for the git plumbing and **lint-staged** for the actual work. The hook is `.husky/pre-commit`:

```
npx lint-staged
```

`package.json` configures lint-staged to format and lint **only staged files**:

```json
"lint-staged": {
  "*.{ts,tsx,js,jsx}": ["prettier --write", "eslint --fix"],
  "*.{json,md,css,html,yml,yaml}": ["prettier --write"]
}
```

Tests do **not** run in the hook. They run in CI and via `make check`. The hook is meant to be sub-second so it's never tempting to skip with `--no-verify`.

`package.json` also has `"prepare": "husky"`, which activates the hook on every `npm install`. New contributors get the hook for free; no manual setup.

When you legitimately need to bypass the hook (rare тАФ committing intentionally unformatted code as a checkpoint, for example), `git commit --no-verify` works.

## ESLint

Flat config in `eslint.config.js`. Rules of note:

- `react-hooks/rules-of-hooks` тАФ strict, so variables prefixed `use*` are treated as hooks. If a non-hook test mock starts with `use`, ESLint will complain.
- `react-hooks/set-state-in-effect` тАФ `setState` synchronously in `useEffect` is an error, not a warning. Use the "adjust state during render" pattern instead (compare prev value, call setter once).
- `react-hooks/refs` тАФ accessing or writing `ref.current` during render is an error. Refs are for effects and event handlers only.
- `react-hooks/exhaustive-deps` тАФ warnings, not errors. We fix them rather than silencing.

`eslint-config-prettier` is the last entry, so Prettier rules win over ESLint stylistic ones (no double-formatting wars).

## Prettier

`.prettierrc` is the default style. `.prettierignore` excludes:

- `node_modules`, `dist`, `coverage`, `package-lock.json` тАФ generated.
- `src/api/schema.ts` тАФ generated by `openapi-typescript`; do not touch.

`make format` runs Prettier then ESLint `--fix` so the two agree.

## TypeScript

Solution-style: root `tsconfig.json` references `tsconfig.app.json` (the `src/` code) and `tsconfig.node.json` (config files). Both children are `composite: true`.

We pin TypeScript at `~5.8.0`. The original plan called for `5.6`; we bumped it because TS 5.6 crashes `tsc -b` on Node 23+ during declaration emit for composite projects with RTK Query types in scope. CI runs Node 20 so it never hit this; local users on Node 23+ would. 5.8 is fine.

If `tsc -b` ever crashes with `Debug Failure. False expression. at getOptionalType`, that's the bug. Bump TypeScript before chasing it any further.

## openapi-typescript

Backend ships `schemas/openapi.yaml` in its repo. We pull it from GitHub raw and emit `src/api/schema.ts`. The full flow is documented in [`../api/client.md`](../api/client.md).

`make gen-api-check` exists for CI to fail if someone forgets to regenerate.

## Vite

`vite.config.ts` covers:

- React plugin (Babel-based, not SWC тАФ SWC was reverted in PR-1 because the Rolldown deprecation noise wasn't worth the speed-up at our size).
- Path alias `@/` тЖТ `src/`.
- `/api` and `/ws` proxies for local dev (see [`../api/client.md`](../api/client.md#base-url-and-the-dev-proxy)).
- `optimizeDeps.include: ['react-use-websocket']` to make Vite's interop behave with that CJS package.
- vitest config: jsdom environment, globals, `vitest.setup.ts` for `@testing-library/jest-dom` matchers.

## Vitest

`make test` runs `vitest run` (one-shot). `make test-watch` for the watch-mode REPL. `make test-cov` adds coverage.

See [`testing.md`](testing.md) for what we mock and how `renderWithProviders` works.
