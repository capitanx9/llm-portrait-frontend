# Architecture

This document is for someone reading the code for the first time. It explains where each thing lives and how a request flows from a button click through to the backend and back.

## Stack

- **React 19** + **TypeScript** + **Vite 8** (Rolldown-based) — the runtime and the build/dev server.
- **MUI v9** + Emotion for components and styling.
- **Redux Toolkit 2** + **RTK Query** for state and data fetching.
- **react-router-dom v7** for routing, **react-use-websocket** for the chat socket.
- **Vitest 4** + Testing Library for tests; **ESLint** + **Prettier** for code quality; **husky** + **lint-staged** for pre-commit formatting.
- **Make** drives day-to-day commands (`make dev`, `make check`, `make gen-api`); see [`development/tooling.md`](development/tooling.md).

**Feature folders own their slice, their RTK Query endpoints, their pages and their tests.** Deleting a feature = deleting its folder. Shared things live one level up: `api/` for the network layer, `app/` for the global singletons (store, theme, router), `components/` for cross-feature UI primitives, `hooks/` for cross-feature hooks.

## Boot sequence

1. Browser hits `/`, gets `index.html`.
2. `<script type="module" src="/src/main.tsx">` loads.
3. `main.tsx` creates the React root inside `#root` and mounts the Provider tree:

   ```
   <Provider store={store}>
     <ThemeProvider theme={theme}>
       <CssBaseline />
       <RouterProvider router={router} />
     </ThemeProvider>
   </Provider>
   ```

4. The auth slice's initial state runs `hydrate()`, which reads `sessionStorage.getItem('llm-portrait.auth')` and seeds `{accessToken, refreshToken, user}` if anything was there.
5. The router matches the URL and renders `<App />` plus the child route.
6. `<App />` reads `auth.accessToken` to decide which AppBar buttons to show (Sign in/Register vs Chat/Profile/Log out).

## Data flow — REST request

A typical authenticated REST call (say `GET /api/chat/rooms/`):

```
component (RoomList)
  └─ useGetRoomsQuery()                 RTK Query hook
       └─ baseApi.endpoints.getRooms.initiate(...)
            └─ baseQueryWithReauth      src/api/baseApi.ts
                 ├─ prepareHeaders sets Authorization: Bearer <accessToken>
                 ├─ fetch('/api/chat/rooms/?limit=100')
                 │     vite dev proxy → http://localhost:8000
                 │     production → same-origin, hits the EC2 backend
                 ├─ if 401:
                 │     POST /api/auth/refresh/ with the stored refresh token
                 │     on 200: dispatch(setCredentials({access}))
                 │              retry the original request
                 │     on fail: dispatch(clearCredentials()) → ProtectedRoute kicks them to /login
                 └─ transformResponse extracts results from the paginated wrapper
```

Components never see fetch directly. They consume `useXxxQuery` / `useXxxMutation` hooks generated by RTK Query and read `data`, `isLoading`, `error` off them.

## Data flow — WebSocket

```
component (ChatPage)
  └─ useWebSocket(socketUrl, { shouldReconnect, reconnectAttempts, reconnectInterval })
       URL = ws(s)://<window.location.host>/ws/chat/<room>/?token=<access>
       dev   = ws://localhost:5173 → vite proxy → ws://localhost:8001 (daphne)
       prod  = wss://<same hostname>
```

Frames in:

- Valid `ChatMessageOut` (`{id, sender, text, created_at}`) → appended to a per-room live buffer kept in the page's state. The buffer is merged with the REST history (deduped by id) before render.
- Error frames (`{error: 'invalid_json' | 'internal', detail}`) → currently ignored on the receive path. Snackbar is reserved for REST errors.

Frames out:

- `{ text }` on every "Send" / Enter. The server saves the row and broadcasts back; the bubble appears via the receive path. **No optimistic update.**

The composer is disabled and a "Reconnecting…" banner is shown whenever `readyState !== OPEN`.

## Auth and the reauth interceptor

The auth slice (`src/features/auth/slice.ts`) keeps `{accessToken, refreshToken, user}` and mirrors them into `sessionStorage` on every write. `baseQueryWithReauth` (`src/api/baseApi.ts`) wraps every REST call:

1. Attach `Authorization: Bearer <access>` if present.
2. On 401, read `refreshToken` from the store.
3. POST it to `/api/auth/refresh/`. If we get a fresh `access`, store it and retry the original request once.
4. If refresh itself fails, clear credentials. `ProtectedRoute` reads `accessToken === null` and bounces the user to `/login`.

`<ProtectedRoute />` wraps `/chat` and `/profile` in the router config. It is a UX safety net — the real gate is the backend's 401 on missing/invalid tokens.

See [`debug/workflow/1-trace-http.md`](debug/workflow/1-trace-http.md) for how to watch the 401 → refresh → retry sequence in Network.

## Types: where they come from

The backend ships `schemas/openapi.yaml` in its repo. `make gen-api` downloads it from GitHub raw and runs `openapi-typescript` over it, writing `src/api/schema.ts`. Endpoint definitions in features pull their request/response shapes out of `components['schemas']` so the frontend can't drift from the backend's contract silently. See [`api/client.md`](api/client.md).

## Tests

- **Unit/integration tests** live next to the code in `*.test.ts(x)` files. We run them with `vitest run` in CI and through `make test` locally.
- We mock `fetch` for RTK Query tests and `react-use-websocket` for the chat page. We use `renderWithProviders` (in `src/test/`) so a component test gets a Redux store and a MemoryRouter without the rest of the app.
- See [`development/testing.md`](development/testing.md) for the patterns we settled on.

## CI / deploy

- Every push and PR runs CI (`make check` minus the dev-only bits): lint, typecheck, vitest, build. See [`development/ci.md`](development/ci.md).
- Production deployment is S3 + CloudFront, driven from `.github/workflows/cd.yml`. The CD wiring lands in a follow-up PR once the AWS account verification clears. See [`deployment/s3-cloudfront.md`](deployment/s3-cloudfront.md).
